# MoM - May 16, 2026

## Key Decisions

### 1. Tool Specification
- Add OpenCode to the discussion instead of "OpenClaw"
- Example use case: demonstrate the extreme spectrum of tool usage patterns

### 2. Stub Strategy
- **Final design approach:** Content mirror + daemon
- **Tool-open trigger strategy rejected:** Replaced with event-based daemon (chokidar)
- **Main conflict resolution concept:** Latest is greatest (based on file modification time)

### 3. Transformers Concept
- Maintain transformers for handling new updates from any vendor
- Support vendors' MCPs, skills, structures, agents, etc.
- Future-proofs against tool evolution

### 4. MVP Scope
- **Single profile only** — no multiple profiles in MVP
- Delay all profile-related discussions for next versions
- Default profile covers all MVP use cases

### 5. Collaboration Feature
- **Decision:** Discard team collaboration from MVP scope
- **Rationale:** Let final users decide what to do with ~/.waslagenie/
- **v1.1+:** Implement team sharing and profile management separately

### 6. MVP Principles

#### Local Agent Creation
- Users create agents in familiar tools (Claude, Gemini, OpenCode)
- Zero-friction workflow — no forced migration to canonical location
- Respect user's tool preferences

#### Daemon Sync
- Event-based file watching (chokidar)
- Handles both current working directory and home directory
- Automatically invoked, low CPU overhead
- Continuously syncs agents visible everywhere

#### Gradual Centralization
- Optional migration to ~/.waslagenie/ for advanced use cases
- Enables future portability and team features
- Not mandatory in MVP

#### Export/Import for Backup
- Individual machine backup/restore
- Enables workflow portability
- Foundation for team sharing in v1.1

#### MCP Re-focus
- MCPs are critical for MVP
- Agents + MCPs cover the highest-pain duplication scenarios
- MCPs require content mirroring (they're JSON configs, not editable by users)

## Implementation Priorities

1. **Scanner & Registry** — Detect tools, discover assets, track state
2. **Adapters** — Tool-specific stub writing (Claude, Gemini, OpenCode)
3. **Syncer** — Orchestrate conflict resolution, stub writing
4. **CLI Commands** — install, sync, status, config, watch
5. **Daemon** — File watcher with auto-sync on changes
6. **Testing** — Validate sync workflows across all tools

## Notes

- **Collaboration feature deferred:** Users can decide how to use ~/.waslagenie/ without built-in team features
- **Daemon always runs:** Once enabled, watches both home and workspace directories automatically
- **Profile system deferred:** Keep MVP simple with single default profile
- **MCP handling:** Requires JSON patching for tools that embed MCP config in settings files
- **Content mirroring:** Primary strategy for MCPs; use for agents only if Option A/C fail
