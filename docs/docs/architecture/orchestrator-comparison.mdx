## Dimension 1 — Agents

### 1.1 Claude Code

<Admonition type="info" title="Format">
Agent definitions are **Markdown files with YAML frontmatter** placed inside
`.claude/agents/` (project-scoped) or `~/.claude/agents/` (user-scoped).
</Admonition>

#### Directory layout

```
# Project-scoped (committed to repo, shared with team)
.claude/
└── agents/
    ├── researcher.md
    ├── code-reviewer.md
    └── test-writer.md

# User-scoped (personal, available across all projects)
~/.claude/
└── agents/
    ├── security-auditor.md
    └── documentation-writer.md
```

#### Agent file anatomy

```markdown
---
name: researcher
description: >
  Deep research agent. Spawned when the user asks to "research", "investigate",
  or "find out about" a topic. Do NOT spawn for code tasks.
tools: Read, Grep, Glob, WebSearch
disallowedTools: Write, Edit, Bash
model: claude-opus-4-6
permissionMode: default
maxTurns: 20
skills:
  - web-research
  - citation-formatter
mcpServers:
  - exa-search
color: purple
---

You are a senior research analyst. Your job is to:
1. Gather information from multiple sources.
2. Synthesize findings into a structured report.
3. Always cite your sources.

Never modify files unless explicitly asked.
```

#### YAML frontmatter field reference

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `name` | `string` | ✅ | Unique identifier. Used in `@agent-name` mentions. |
| `description` | `string` | ✅ | **Critical** — Claude uses this to decide which agent to spawn. Be specific. |
| `tools` | `string` (CSV) | ❌ | Allowlist. When set, agent can ONLY use these tools. |
| `disallowedTools` | `string` (CSV) | ❌ | Denylist. Applied before `tools`. |
| `model` | `string` | ❌ | Defaults to session model. Accepts any Claude model string. |
| `permissionMode` | `string` | ❌ | `default` \| `acceptEdits` \| `bypassPermissions` |
| `maxTurns` | `number` | ❌ | Cap on agentic loop iterations. |
| `skills` | `string[]` | ❌ | Skills to pre-load. Must match skill names in `.claude/skills/`. |
| `mcpServers` | `string[]` | ❌ | Additional MCP servers the agent connects to at spawn time. |
| `hooks` | `object` | ❌ | Lifecycle hooks. Not supported in plugin sub-agents. |
| `initialPrompt` | `string` | ❌ | First turn injected when spawning. |
| `memory` | `boolean` | ❌ | Whether agent has access to memory tools. |
| `effort` | `string` | ❌ | `low` \| `medium` \| `high` |
| `background` | `boolean` | ❌ | Run in background (non-blocking). |
| `isolation` | `boolean` | ❌ | Isolated context window from parent. |
| `color` | `string` | ❌ | UI label color. |

#### Lifecycle — how Claude Code discovers and runs agents

```
Session start
│
├─ Scans .claude/agents/ (workspace) → lower precedence
├─ Scans ~/.claude/agents/ (user) → higher precedence  
│   └─ Also loads managed agents from .claude/agents/ inside
│      managed settings directory (org-controlled, highest precedence)
│
├─ Agent names + descriptions injected into system prompt
│
User message / Agent tool call
│
├─ Claude matches intent → selects agent
├─ Spawns isolated context window
├─ Agent runs its own agentic loop (gather → act → verify)
│   └─ Subagents CANNOT spawn further subagents
├─ Result compressed → returned to parent context
│
Session end → all sub-contexts destroyed
```

#### Multi-agent routing

```markdown
<!-- In CLAUDE.md or agent body -->
## Agent Team

- @researcher — for any "find out" or "research" task
- @code-reviewer — for PRs and code quality checks
- @test-writer — for writing or updating test suites

Lead agent coordinates. Each specialist runs in isolation.
```

#### WaslaGenie adapter checklist — Claude Code agents

- [ ] Scan `.claude/agents/*.md` (workspace) and `~/.claude/agents/*.md` (user)
- [ ] Parse YAML frontmatter: extract `name`, `description`, `tools`, `model`
- [ ] Treat `name` as the canonical stub key across tools
- [ ] When writing a stub: preserve frontmatter fields as-is, add `waslagenie_ref` + `origin`
- [ ] Precedence on conflict: user file wins over workspace file
- [ ] Do NOT sync `hooks` or `permissionMode` — these are Claude-specific and may cause permission escalation in other tools

### 1.2 Gemini CLI

<Admonition type="warning" title="Architecture difference">
Gemini CLI does **not** use discrete per-agent Markdown files for spawning
sub-agents the way Claude Code does. Agent context is delivered through
**context files** (`GEMINI.md`) and **skills**. Sub-agent spawning is not
a native primitive — parallel specialization is achieved via skills and
session routing.
</Admonition>

#### Directory layout

```
# Project-scoped context (shared with team)
.gemini/
├── settings.json        ← MCP + model config
└── skills/              ← On-demand skill library

# User-scoped context (personal)
~/.gemini/
├── settings.json        ← Global MCP + preferences
└── skills/              ← Global skill library

# Interop alias (cross-tool compatible)
.agents/
└── skills/              ← Same as .gemini/skills/ — higher precedence
```

#### Context file — `GEMINI.md`

`GEMINI.md` is the primary "agent instruction" surface in Gemini CLI. It
is loaded unconditionally at session start, unlike skills (which are
on-demand).

```markdown
# Project Context

## Stack
- TypeScript monorepo, Node 22, Vitest, Turborepo

## Behaviour
- Always confirm destructive operations before executing
- Use the filesystem MCP tool for all file writes
- When asked to review, trigger the `code-reviewer` skill

## Agent Routing
- Research tasks → use web-search skill
- DB tasks → use the postgres MCP tool
- Deployment → use the github-actions skill
```

#### Lifecycle — how Gemini CLI handles "agents"

```
Session start
│
├─ Loads GEMINI.md (workspace root, then user home)
├─ Loads all skill metadata (name + description)
│   from ~/.gemini/skills/, .gemini/skills/, .agents/skills/
│   (see Skills section for full precedence order)
│
User message
│
├─ Gemini matches intent to a skill or built-in tool
├─ If skill matches → calls activate_skill tool
│   └─ User sees consent prompt (name, purpose, path)
│   └─ Skill body + folder injected into conversation history
│   └─ Skill directory added to allowed file paths
│
├─ Tools execute inline (no separate context window spawned)
│   └─ No native sub-agent isolation — all runs in same session
│
Session end → no persistent sub-context state
```

#### WaslaGenie adapter checklist — Gemini CLI agents

- [ ] Scan `GEMINI.md` in workspace and `~/.gemini/GEMINI.md` for agent routing instructions
- [ ] There are **no discrete agent files** to sync — agent identity lives in skills + GEMINI.md
- [ ] When receiving a Claude agent stub, map `description` field into a new skill's `description` frontmatter
- [ ] Do NOT attempt to write `.gemini/agents/*.md` — this path is not natively supported
- [ ] Log a warning in the registry if a Claude agent has `permissionMode: bypassPermissions` — no Gemini equivalent exists

### 1.3 OpenClaw

<Admonition type="info" title="Architecture">
OpenClaw uses a **workspace file model**. Agent personas, routing rules, memory
locations, and skill assignments are all defined inside `AGENTS.md` (and
companion files `SOUL.md`, `MEMORY.md`) within the agent workspace directory,
which defaults to `~/.openclaw/workspace/`.
</Admonition>

#### Directory layout

```
# Agent workspace (default location)
~/.openclaw/workspace/
├── AGENTS.md        ← agent definitions + routing table
├── SOUL.md          ← personality, tone, preferences
├── MEMORY.md        ← long-term memory (auto-updated)
└── HEARTBEAT.md     ← proactive task scheduler

# Multi-agent setup (each agent has its own workspace)
~/.openclaw/workspaces/
├── writer/
│   ├── AGENTS.md
│   └── skills/
├── researcher/
│   ├── AGENTS.md
│   └── skills/
└── coder/
    ├── AGENTS.md
    └── skills/

# Config and state
~/.openclaw/
├── openclaw.json    ← main config (MCP, skills, agents)
├── skills/          ← managed/global skills
└── tools/           ← skill-installed binaries
```

#### `AGENTS.md` anatomy

```markdown
# Agent Routing

## Task routing table

| Task type        | Context files to load     | Skills to activate        | Skip                  |
|------------------|---------------------------|---------------------------|-----------------------|
| writing          | writing-room.md           | humanizer, fact-checker   | code-context.md       |
| code-review      | code-standards.md         | code-reviewer, linter     | writing-room.md       |
| research         | —                         | web-search, citation      | —                     |
| deployment       | devops-context.md         | github-actions            | writing-room.md       |

## Agent definitions

### writer
Specialist for long-form content. Load for: blog posts, documentation,
READMEs, release notes. Skills: humanizer, fact-checker.
Do not use for code tasks.

### coder
Specialist for implementation. Load for: feature development, bug fixing,
refactoring. Skills: code-reviewer, test-writer, linter.
Access to: filesystem MCP, github MCP.
```

#### Multi-agent config in `openclaw.json`

```json5
{
  agents: {
    defaults: {
      skills: ["github", "weather"],
      sandbox: {
        enabled: true
      }
    },
    list: [
      { id: "writer", skills: ["humanizer", "fact-checker"] },
      { id: "coder", skills: ["code-reviewer", "linter", "test-writer"] },
      { id: "researcher" }  // inherits defaults
    ]
  }
}
```

#### Lifecycle — how OpenClaw routes agents

```
Session start
│
├─ Reads ~/.openclaw/openclaw.json → builds agent list
├─ Loads AGENTS.md from active workspace
├─ Snapshots eligible skills per agent (from allowlist)
│
User message
│
├─ Channel routing → determines which agent receives message
├─ Agent loads its context: SOUL.md, MEMORY.md, assigned skills
├─ Agent runs in agentic loop (gather → act → verify → repeat)
│
Skills fire mid-session (hot reload if watcher is on)
│
Session end
├─ MEMORY.md updated (if memory engine active)
└─ Skill snapshot invalidated → rebuilt on next session
```

#### WaslaGenie adapter checklist — OpenClaw agents

- [ ] Scan `~/.openclaw/workspace/AGENTS.md` for agent definitions
- [ ] Parse routing table: extract agent names and their skill assignments
- [ ] When writing a Claude agent as an OpenClaw stub: append to `AGENTS.md` routing table rather than writing a separate file
- [ ] Map Claude's `tools` allowlist → OpenClaw's `agents.list[].skills` allowlist
- [ ] `SOUL.md` and `MEMORY.md` are **not syncable** — mark as `origin-only` in registry
- [ ] When syncing FROM OpenClaw TO Claude Code: generate one `.claude/agents/<name>.md` per agent definition row in `AGENTS.md`

## Dimension 2 — MCP Servers

### 2.1 Claude Code

#### Config file locations and precedence

```
# Workspace-scoped (committed to repo)
.mcp.json                         ← primary workspace MCP config

# User-scoped (personal, not committed)
~/.claude/settings.json           ← user-level MCP config (mcpServers key)

# Project settings (alternative workspace location)
.claude/settings.json             ← project settings (also accepts mcpServers)

# Managed (org-controlled, highest precedence)
.claude/settings.managed.json
```

Precedence: `settings.managed.json` > `~/.claude/settings.json` > `.mcp.json` / `.claude/settings.json`

#### `.mcp.json` format

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"],
      "env": {
        "NODE_ENV": "production"
      }
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    },
    "remote-server": {
      "url": "https://mcp.example.com/sse",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}
```

#### Transport options

| Transport | Config key | Use case |
|-----------|-----------|----------|
| stdio | `command` + `args` | Local subprocess. Most common. |
| SSE | `url` | Remote server, Server-Sent Events. |
| HTTP | `url` (http/https) | Remote server, standard HTTP. |

#### Tool naming

Tools exposed by MCP servers are available as-is by their registered name within
the Claude session. No prefix is applied (unlike Gemini CLI).

```
# Tool from "github" MCP server
create_issue
list_pull_requests
get_repository

# Tool from "filesystem" MCP server
read_file
write_file
list_directory
```

#### Watcher behavior

- MCP servers are started at session open and killed at session close.
- Config changes require a session restart to take effect.
- Per-agent MCP assignment is supported via the `mcpServers` field in agent frontmatter.

#### WaslaGenie adapter checklist — Claude Code MCP

- [ ] Scan `.mcp.json` at project root, then `.claude/settings.json`, then `~/.claude/settings.json`
- [ ] Parse `mcpServers` object: extract server name + transport + command/url
- [ ] Do NOT sync `env` values that contain `${VAR}` templates — flag as `env-dependent` in registry
- [ ] When writing a stub for a Gemini or OpenClaw MCP: translate to `.mcp.json` format
- [ ] Treat server name (object key) as the canonical stub key

### 2.2 Gemini CLI

#### Config file locations and precedence

```
# Workspace-scoped
.gemini/settings.json             ← project MCP config

# User-scoped
~/.gemini/settings.json           ← global MCP config

# Admin-managed (enterprise)
Managed via policy — injected into effective config
```

Precedence: admin-managed > workspace > user

#### `.gemini/settings.json` format

```json
{
  "mcpServers": {
    "github": {
      "command": "docker",
      "args": [
        "run", "-i", "--rm",
        "-e", "GITHUB_PERSONAL_ACCESS_TOKEN",
        "ghcr.io/github/github-mcp-server"
      ]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user"],
      "env": {
        "NODE_ENV": "production"
      }
    },
    "remote-exa": {
      "url": "https://mcp.exa.ai/mcp",
      "httpUrl": "https://mcp.exa.ai/mcp"
    }
  },
  "theme": "Default",
  "selectedAuthType": "oauth-personal"
}
```

#### Transport options

| Transport | Config key | Precedence |
|-----------|-----------|-----------|
| HTTP | `httpUrl` | Highest |
| SSE | `url` | Middle |
| stdio | `command` | Lowest |

When multiple transport keys are set, `httpUrl` wins, then `url`, then `command`.

#### Tool naming — fully qualified names

Gemini CLI **prefixes all MCP tools** with `mcp_<serverAlias>_`:

```
# Tool from "github" server
mcp_github_create_issue
mcp_github_list_pull_requests

# Tool from "filesystem" server
mcp_filesystem_read_file
mcp_filesystem_write_file
```

This means the same MCP server connected to both Claude Code and Gemini CLI
will expose tools under **different names**. WaslaGenie must handle this when
writing `GEMINI.md` instructions or skill bodies that reference tool names.

#### Admin controls

```json
{
  "mcpServers": { ... },
  "enableMcpServers": true,
  "adminMcpServers": {
    "required-logging": {
      "command": "npx",
      "args": ["-y", "@corp/logging-mcp"]
    }
  }
}
```

- `enableMcpServers: false` — disables all MCP for the session
- `adminMcpServers` — servers always injected regardless of user config
- Extensions (GitHub, Figma, Google Workspace, Exa) are installed as MCP-backed integrations

#### WaslaGenie adapter checklist — Gemini CLI MCP

- [ ] Scan `.gemini/settings.json` (workspace), then `~/.gemini/settings.json` (user)
- [ ] Parse `mcpServers` key — same structure as Claude Code's format
- [ ] Note transport precedence: `httpUrl` > `url` > `command`
- [ ] When generating tool-reference docs/skills for Gemini: prepend `mcp_<alias>_` to all tool names
- [ ] Skip `adminMcpServers` during sync — org-managed, not user-owned
- [ ] Flag `enableMcpServers: false` in registry — mark all MCPs as `disabled` for this tool

### 2.3 OpenClaw

#### Config file locations

```
# Primary config (all MCP config lives here)
~/.openclaw/openclaw.json

# No workspace-scoped MCP config file
# (workspace isolation is handled via agents.list, not separate files)
```

#### `openclaw.json` — MCP section

```json5
{
  mcp: {
    servers: {
      filesystem: {
        command: "npx",
        args: ["-y", "@modelcontextprotocol/server-filesystem", "/Users/you/Documents"],
        description: "Access to Documents folder"
      },
      github: {
        command: "npx",
        args: ["-y", "@modelcontextprotocol/server-github"],
        env: {
          GITHUB_PERSONAL_ACCESS_TOKEN: "ghp_..."
        }
      },
      postgres: {
        url: "https://mcp.example.com/postgres/sse",
        description: "Production read-only DB"
      }
    }
  }
}
```

#### Key structural differences from Claude Code / Gemini CLI

| Aspect | Claude Code | Gemini CLI | OpenClaw |
|--------|-------------|------------|----------|
| Config key path | `mcpServers` | `mcpServers` | `mcp.servers` |
| Config file | `.mcp.json` or `settings.json` | `.gemini/settings.json` | `openclaw.json` |
| Workspace vs user | Two separate files | Two separate files | Single file (user-scoped only) |
| Per-agent MCP | Via agent frontmatter | Via skill configuration | Via `agents.list[].mcpServers` |
| MCP install helper | `claude mcp add <server>` | `/mcp add <server>` | `openclaw mcp install <slug>` (also `mcp-bridge` skill) |

#### Per-agent MCP scoping in OpenClaw

```json5
{
  agents: {
    list: [
      {
        id: "coder",
        mcpServers: ["filesystem", "github"]
        // Only these two MCP servers are available to the coder agent
      },
      {
        id: "researcher"
        // Inherits no default MCP — uses web-search skill instead
      }
    ]
  }
}
```

#### MCP via `mcp-bridge` skill

OpenClaw also exposes an alternative path: the `mcp-bridge` skill (installed
from ClawHub) wraps MCP discovery and tool routing inside a skill, enabling
MCP-like behavior without direct `openclaw.json` config.

```bash
openclaw skills install mcp-bridge
```

This is separate from native `mcp.servers` config and is used for dynamic
tool discovery without a restart.

#### WaslaGenie adapter checklist — OpenClaw MCP

- [ ] Scan `~/.openclaw/openclaw.json`, read `mcp.servers` key (note: nested under `mcp`, not top-level)
- [ ] Translate `mcp.servers.<name>` → `mcpServers.<name>` when writing Claude/Gemini stubs
- [ ] Do NOT attempt to write a workspace-scoped MCP file for OpenClaw — it doesn't exist
- [ ] When syncing MCP env vars: mask secret values, store only key names in registry
- [ ] If `mcp-bridge` skill is detected, log it as a secondary MCP source — do not parse its internal config

## Dimension 3 — Skills

### 3.1 The SKILL.md Open Standard

All three orchestrators implement the **Agent Skills open standard** (originally
authored by Anthropic, Dec 2025). The format is **100% portable across tools**.

#### Universal skill directory structure

```
<skill-name>/
├── SKILL.md          ← required — definition, frontmatter, instructions
├── scripts/          ← optional — shell/Python scripts the agent can run
│   ├── check.sh
│   └── validate.py
├── references/       ← optional — documents, templates, examples
│   ├── style-guide.md
│   └── api-reference.json
└── examples/         ← optional — input/output examples for the agent
    ├── input.md
    └── output.md
```

Rules:
- `SKILL.md` must be at the **root** of the skill directory (one level deep from the skills folder)
- Files nested more than one directory deep are NOT auto-discovered
- Directory name = skill name (must match `name:` in frontmatter)
- `SKILL.md` filename is **case-sensitive** on Linux/macOS (strict case required)

#### Universal `SKILL.md` frontmatter

```markdown
---
name: code-reviewer          # required — unique identifier, matches directory name
description: >               # required — CRITICAL: this triggers skill activation
  Reviews code changes for correctness, security, and style. Use when the user
  asks to "review" code, a PR, or wants feedback on an implementation.
---

# Code Review Skill

You are a senior software engineer specialized in code quality.

## When this skill activates
- User says "review", "check", "audit", or "look over" code
- A pull request is mentioned

## Process
1. Read all changed files
2. Check for: correctness, edge cases, security issues, performance
3. Rate each finding: CRITICAL / HIGH / MEDIUM / LOW
4. Suggest specific fixes with code examples

## Rules
- Never approve code with CRITICAL findings
- Always provide actionable feedback, not just observations
```

### 3.2 Claude Code — Skills

#### Directory layout and precedence

```
# Workspace-scoped (project, version-controlled)
.claude/skills/
├── code-reviewer/
│   ├── SKILL.md
│   └── scripts/
│       └── eslint-check.sh
├── test-writer/
│   └── SKILL.md
└── api-designer/
    ├── SKILL.md
    └── references/
        └── openapi-spec.yaml

# User-scoped (personal, all projects)
~/.claude/skills/
├── security-auditor/
│   └── SKILL.md
└── commit-msg/
    └── SKILL.md

# Interop alias (cross-tool compatible path)
.agents/skills/     ← same as .claude/skills/
~/.agents/skills/   ← same as ~/.claude/skills/
```

Precedence: workspace > user

#### Lifecycle

```
Session start
│
├─ Scans ~/.claude/skills/ + .claude/skills/
│   └─ Injects metadata (name + description) into system prompt
│   └─ Context cost: ~195 chars base + ~97 chars per skill
│
User message
│
├─ Claude reads description → decides if skill is relevant
├─ Claude calls read_me tool on SKILL.md → full instructions loaded
│   └─ Progressive disclosure: body only loaded when needed
│   └─ Skill directory added to allowed file paths
│
Skill executes inline within current session context
│
Session end → skill context cleared
```

#### Integration with agents

#### Trust requirement

- **Workspace skills** (`.gemini/skills/` and `.agents/skills/`) require the workspace to be **trusted**
- Run `/trust` in a Gemini CLI session and restart to enable workspace skills
- **User skills** (`~/.gemini/skills/`) are always trusted — no trust step needed

#### Extended SKILL.md frontmatter (OpenClaw-specific)

OpenClaw supports additional frontmatter keys beyond the base standard:

```markdown
---
name: image-lab
description: >
  Generate or edit images via provider-backed workflow. Use when the user
  asks to create, generate, or modify an image.
homepage: https://clawhub.ai/skills/image-lab
user-invocable: true
disable-model-invocation: false
command-dispatch: tool
command-tool: image_generate
metadata: {"openclaw":{"emoji":"🎨","requires":{"bins":["uv"],"env":["GEMINI_API_KEY"],"config":["browser.enabled"]},"primaryEnv":"GEMINI_API_KEY","install":[{"id":"brew","kind":"brew","formula":"image-tools","bins":["image-lab"]}]}}
---
```

| Field | Default | Notes |
|-------|---------|-------|
| `homepage` | — | URL shown in macOS Skills UI |
| `user-invocable` | `true` | Exposes skill as user slash command |
| `disable-model-invocation` | `false` | `true` = available via slash but not model-triggered |
| `command-dispatch` | — | `tool` = bypass model, dispatch directly to tool |
| `command-tool` | — | Tool name for direct dispatch |
| `metadata` | — | **Single-line JSON object** — gates, installer specs, emoji |

#### Gating (`metadata.openclaw.requires`)

```markdown
metadata: {"openclaw":{"requires":{"bins":["uv","git"],"env":["API_KEY"],"config":["browser.enabled"]}}}
```

| Gate | What it checks |
|------|---------------|
| `bins` | Binary exists on `PATH` (host) |
| `anyBins` | At least one binary exists |
| `env` | Env var is set |
| `config` | Key in `openclaw.json` is truthy |

Skills that fail their gates are silently excluded from the session prompt.

#### Per-agent skill allowlists

```json5
{
  agents: {
    defaults: {
      skills: ["github", "web-search"]
    },
    list: [
      { id: "writer", skills: ["humanizer", "fact-checker"] }, // overrides defaults
      { id: "coder" },                                          // inherits defaults
      { id: "locked-down", skills: [] }                        // no skills
    ]
  }
}
```

An agent with a non-empty `skills` array replaces the defaults entirely —
there is **no merge**.

#### Skill watcher (hot reload)

## Appendix — Common Gotchas for Adapter Authors

| Gotcha | Tool(s) | Mitigation |
|--------|---------|-----------|
| MCP `env` contains secret values | All | Store only key names in registry; never mirror values |
| Gemini tool names are prefixed (`mcp_alias_tool`) | Gemini | Translate when generating skill bodies that reference tool names |
| OpenClaw `metadata` must be single-line JSON | OpenClaw | Serialize to one line before writing; do not pretty-print |
| Workspace skills in Gemini need `/trust` | Gemini | Flag in registry; add note to stub |
| `permissionMode: bypassPermissions` in Claude | Claude | Do not propagate to other tools — drop this field in stubs |
| OpenClaw agent allowlists do not merge with defaults | OpenClaw | When writing allowlist, write the complete final list |
| Claude subagents cannot spawn further subagents | Claude | Flatten nested agent trees before syncing |
| Skills watcher in OpenClaw invalidates snapshot mid-session | OpenClaw | Trigger re-scan after writing skill stubs |
| `.openclaw/skills/` is NOT a valid path | OpenClaw | The managed skills path is `~/.openclaw/skills/` (user-scoped only) |
| SKILL.md file name is case-sensitive on Linux | All | Always write uppercase `SKILL.md` |
